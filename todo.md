# myproxy 改进任务清单

本文档用于跟踪下一阶段的改进工作，重点覆盖三件事：

1. 发掘并整理新需求
2. 增加火焰图与内存使用诊断能力
3. 围绕项目真实使用场景补齐稳定性与验证

## 一、需求梳理

### 1. 资源与诊断能力需求

- [ ] 明确“诊断中心”功能范围：是否包含内存、goroutine、日志、节点状态、订阅状态。
- [x] 明确诊断入口位置：设置页新增“诊断”菜单，还是复用现有“日志”区域扩展。
- [ ] 明确诊断功能面向对象：开发者专用、普通用户可见、还是通过配置开关控制。
- [x] 明确是否允许用户在 UI 内直接开启/关闭 `pprof`。
- [ ] 明确导出诊断文件的保存目录、命名规则、保留策略。

### 2. 长期运行需求

- [ ] 定义“长期运行”的目标场景：连续运行 8 小时、24 小时、72 小时。
- [ ] 定义内存增长告警阈值：例如 30 分钟持续上涨、切换节点后不回落。
- [ ] 定义 goroutine 数量告警阈值。
- [ ] 定义日志文件大小和归档策略是否需要进一步增强。
- [ ] 明确系统休眠/唤醒后是否需要自动恢复状态。

### 3. 代理运行与节点切换需求

- [ ] 定义节点切换后的验收条件：旧实例释放、状态更新、系统代理一致。
- [ ] 定义启动失败时的错误分类展示：配置错误、端口占用、节点不可达、xray 启动失败。
- [ ] 明确是否需要自动重试、自动回退上一个节点。
- [ ] 明确是否需要记录节点切换耗时与最近一次失败原因。

### 4. 订阅与大数据量需求

- [ ] 定义大订阅场景：100、500、1000 节点时的 UI 与内存要求。
- [ ] 明确订阅更新时是否需要显示内存峰值和处理耗时。
- [ ] 明确订阅更新失败后是否保留旧数据并提示差异。
- [ ] 明确批量更新期间是否需要禁用部分 UI 操作，避免重复触发。

## 二、架构与设计任务

### 1. 诊断模块设计

- [x] 设计新的诊断页面结构，遵守 `ui -> service -> store -> database` 分层规则。
- [x] 确定是否新增 `internal/service/diagnostics.go` 负责采样、导出和状态汇总。
- [x] 确定是否新增 `internal/model/diagnostic_snapshot.go` 或类似模型，统一承载运行时诊断数据。
- [ ] 明确哪些诊断数据只驻留内存，哪些需要落盘。
- [ ] 设计诊断能力与现有 `LogsPanel`、`TrafficChart` 的关系，避免职责混杂。

### 2. pprof 能力收敛

- [x] 将 `cmd/gui/main.go` 中的临时 `pprof` 启动逻辑迁移到独立模块，不在入口直接裸起 goroutine。
- [x] 设计 `PprofService` 或 `DiagnosticsService`，统一管理启动、关闭、状态查询。
- [x] 增加配置项：`debugPprofEnabled`。
- [x] 增加配置项：`debugPprofAddr`。
- [x] 明确默认值：默认关闭，仅手动启用。
- [x] 增加安全约束：仅监听 `127.0.0.1` 或 `localhost`。

### 3. 诊断文件管理

- [x] 设计诊断文件目录，例如 `data/diagnostics/`。
- [x] 设计导出文件命名格式：`heap_YYYYMMDD_HHMMSS.pprof`。
- [x] 设计火焰图文件命名格式：`heap_YYYYMMDD_HHMMSS.svg` 或 `.html`。
- [ ] 增加文件清理策略，避免诊断文件无限增长。
- [ ] 明确导出失败时的错误提示和日志记录格式。

## 三、后端任务

### 1. 运行时采样

- [x] 新增运行时采样逻辑，周期读取 `runtime.ReadMemStats`。
- [x] 采集指标至少包含：`Alloc`、`HeapAlloc`、`HeapInuse`、`Sys`、`NumGC`、`Goroutines`。
- [x] 为采样逻辑增加可控生命周期，避免自身产生 goroutine 泄漏。
- [x] 设计采样周期，建议支持 1s、5s、10s 三档。
- [x] 限制内存中的历史采样点数量，避免趋势图自身成为新的内存来源。

### 2. 快照导出

- [x] 增加“导出堆内存快照”接口。
- [x] 增加“导出 goroutine 快照”接口。
- [x] 增加“导出诊断摘要”接口，输出文本或 JSON，记录版本、时间、节点、代理状态。
- [x] 导出操作需统一走 Service 层，不在 UI 中直接操作文件系统。
- [x] 导出完成后返回文件路径，供 UI 打开或提示。

### 3. 火焰图生成

- [x] 评估火焰图生成方案：调用 `go tool pprof` 还是仅导出 profile 文件。
- [x] 如果采用外部生成，封装命令调用与错误处理。
- [x] 如果采用 SVG/HTML 输出，确认目标格式在各平台可打开。
- [x] 生成失败时保留原始 `.pprof` 文件，避免排障信息丢失。
- [ ] 记录火焰图生成耗时，避免 UI 卡死。

### 4. 运行状态汇总

- [x] 增加统一状态摘要接口，供 UI 展示当前代理状态、节点名称、运行时指标。
- [x] 增加最近一次节点切换时间记录。
- [x] 增加最近一次订阅更新时间记录。
- [x] 增加最近一次诊断导出结果记录。

## 四、UI 任务

### 1. 设置页扩展

- [x] 在设置页新增“诊断”菜单项。
- [x] 设计诊断页布局，避免固定像素间距，统一使用 Fyne 主题间距。
- [x] 展示当前运行概览：代理状态、当前节点、内存、goroutine、GC 次数。
- [x] 增加最近 5 分钟或 15 分钟的内存趋势图。
- [x] 增加最近 5 分钟或 15 分钟的 goroutine 趋势图。

### 2. 操作入口

- [x] 增加“导出堆快照”按钮。
- [x] 增加“导出 goroutine 快照”按钮。
- [x] 增加“生成火焰图”按钮。
- [x] 增加“打开诊断目录”按钮。
- [x] 增加“复制诊断摘要”按钮。

### 3. 状态与反馈

- [ ] 导出进行中显示禁用态或加载态，避免重复点击。
- [ ] 导出成功后显示文件路径和时间。
- [x] 导出失败后显示中文错误信息，并写入日志。
- [ ] 采样未启动时显示明确提示，而不是空白区域。
- [x] 当 `pprof` 未开启时，相关按钮和提示状态要一致。

### 4. 趋势图与火焰图展示

- [ ] 评估是否复用现有 `TrafficChart` 思路实现内存趋势图。
- [x] 如复用图表组件，先抽象通用趋势图组件，避免把流量图继续堆功能。
- [ ] 确认火焰图是内嵌展示、外部打开，还是仅提供文件路径。
- [ ] 如果内嵌展示成本过高，第一期先做“导出 + 打开文件”。

## 五、稳定性与资源释放专项

### 1. 现有模块资源治理

- [x] 检查 `LogsPanel`、`TrafficChart`、页面监听器、托盘、xray 生命周期是否统一受 `AppState.Cleanup()` 管理。
- [ ] 继续排查是否还有未解绑的 binding listener。
- [ ] 继续排查是否还有后台 ticker、timer、watcher 未统一关闭。
- [ ] 继续排查主题切换、页面重建后是否有旧对象残留引用。

### 2. 代理与诊断交互

- [ ] 验证频繁启停代理时，诊断采样逻辑是否稳定。
- [ ] 验证切换节点时，趋势图是否出现异常跳变或错误引用旧实例。
- [ ] 验证停止代理后，诊断页状态是否及时刷新。
- [ ] 验证退出应用时，诊断采样 goroutine 是否全部退出。

## 六、场景化测试任务

### 1. 常规用户场景

- [ ] 启动应用，连接代理，最小化到托盘，运行 1 小时，观察内存趋势。
- [ ] 日常切换 10 次节点，确认内存和 goroutine 没有阶梯式上涨。
- [ ] 打开/关闭设置页、日志页、诊断页多次，确认无明显增长。

### 2. 长期驻留场景

- [ ] 持续运行 8 小时，记录内存曲线。
- [ ] 持续运行 24 小时，记录内存曲线与 goroutine 数量。
- [ ] 期间执行订阅更新、节点切换、托盘显示/隐藏，验证资源是否稳定。

### 3. 订阅大数据场景

- [ ] 导入大订阅并更新，记录峰值内存。
- [ ] 订阅更新完成后观察内存是否回落到稳定区间。
- [ ] 节点数较大时检查 NodePage 刷新性能和 UI 卡顿情况。

### 4. 开发排障场景

- [ ] 在代理空闲、代理高流量、频繁切节点三种状态下分别导出 heap 快照。
- [ ] 对比不同时间点的 heap 快照，确认是否存在持续增长对象。
- [ ] 用火焰图确认主要内存来源来自 UI、日志、xray 还是订阅解析。
